kafka是什么？

多分区、多副本且基于ZooKeeper的分布式消息系统。

角色：

  1. 消息系统 可以保证消息顺序性和回溯消费的消息系统

  2. 存储系统 消息持久化到磁盘

  3. 流式处理平台

基本概念

基本结构： producer -> Broker -> consumer，元数据存储在ZooKeeper集群。

主题 topic

​ 生产者指定发送、消费者订阅消费。

分区 partition

  1. 一个分区只属于单个主题

  2. 同一个主题下的不同分区包含的消息不同

  3. 分区存储层面上类似可追加的日志，用偏移量保证分区内的顺序性（分区有序而不是主题有序）

  4. 分区可以分布在不同的broker上，即一个主题可以横跨多个broker.

  5. 多副本机制，只有leader副本负责交互，follower副本负责同步。

    1. 同步机制：leader维护两个参数HW 和 LEO ,

      HW高水位表示当前消费者可消费的（n+1）位置(全部副本同步完成)，LEO表示最新消息（n+1）位置（可能follower未同步完成）。

    2. 概念： ISR(In-Sync Replicas) 表示已同步完成的，OSR表示未同步完成的。AR=ISR + OSR

安装与启动

先启动ZooKeeper, 再启动kafka.

基本流程

服务端参数配置

生产者：配置参数——>创建生产者客户端实例——>构建消息——>发送 ——>关闭客户端

消费者：配置参数——>设置消费组名称——> 创建消费者客户端实例——>订阅主题——> 循环消费消息

二叉树解题的思维模式分两类：

1、是否可以通过遍历一遍二叉树得到答案？如果可以，用一个 traverse 函数配合外部变量来实现，这叫「遍历」的思维模式。

2、是否可以定义一个递归函数，通过子问题（子树）的答案推导出原问题的答案？如果可以，写出这个递归函数的定义，并充分利用这个函数的返回值，这叫「分解问题」的思维模式。

无论使用哪种思维模式，你都需要思考：

如果单独抽出一个二叉树节点，它需要做什么事情？需要在什么时候（前/中/后序位置）做？其他的节点不用你操心，递归函数会帮你在所有节点上执行相同的操作。

操作系统

操作系统基本概念

  1. 操作系统管理计算机硬件和软件资源的程序，屏蔽了硬件层的复杂性。可理解为人机之间的接口。负责文件管理、进程管理、设备管理、内存管理。

  2. 进程在系统上的运行分为用户态和系统态（内核态）。用户态权限低，系统态要求更高权限。从用户态申请转到内核态的场景，包括系统调用、发生异常、外部设备中断。用户态转内核态时，要保存用户态的寄存器信息，返回时恢复，开销较大。

  3. 进程和线程的区别：
    进程是计算机中的程序关于某数据集合上的一次运行活动，是系统进行资源分配和调度的基本单位。线程是更小的运行单位，是CPU调度和分派的基本单位。一个进程可以产生多个线程，线程共享进程内存空间，线程也有自己的空间（程序计数器、虚拟机栈、本地方法栈）

  4. 进程状态包括 创建（new）-> 就绪(ready) -> 运行(running) -> 阻塞(waiting) ->结束（terminated）（正常或中断）

  5. 进程间通信
    概念：进程间相互独立，数据无法直接交换，所以要找中间人内核。在内核里面开辟缓冲区，进程间通过缓冲区进行通信。

  2. tcp三次握手和四次挥手过程
    三次握手才可以阻止重复历史连接的初始化（主要原因）
    三次握手才可以同步双方的初始序列号
    三次握手才可以避免资源浪费

    保证客户端发送的最后一个ack报文段能够到达服务器。因为这最后一个ack确认包可能会丢失，然后服务器就会超时重传第三次挥手的fin信息报，然后客户端再重传一次第四次挥手的ack报文。如果没有这2msl，客户端发送完最后一个ack数据报后直接关闭连接，那么就接收不到服务器超时重传的fin信息报(此处应该是客户端收到一个非法的报文段，而返回一个RST的数据报，表明拒绝此次通信，然后双方就产生异常，而不是收不到。)，那么服务器就不能按正常步骤进入close状态。那么就会耗费服务器的资源。当网络中存在大量的timewait状态，那么服务器的压力可想而知。

    原因2：在第四次挥手后，经过2msl的时间足以让本次连接产生的所有报文段都从网络中消失，这样下一次新的连接中就肯定不会出现旧连接的报文段了。也就是防止我们上一篇文章 为什么tcp是三次握手而不是两次握手？ 中说的：已经失效的连接请求报文段出现在本次连接中。如果没有的话就可能这样：这次连接一挥手完马上就结束了，没有timewait。这次连接中有个迷失在网络中的syn包，然后下次连接又马上开始，下个连接发送syn包，迷失的syn包忽然又到达了对面，所以对面可能同时收到或者不同时间收到请求连接的syn包，然后就出现问题了。

数据库

  1. 三范式
    数据库第一二三范式到底在说什么？ - 刘慰的文章 - 知乎
    https://zhuanlan.zhihu.com/p/20028672

    • 1NF
      一个字段表示一个值。不能再分解。比如进货单价和进货数量 必须分开两个字段。

    • 2NF
      部分函数依赖,2NF在1NF的基础之上，消除了非主属性对于码的部分函数依赖。即假设选择码(A, B）- > C ，则不能直接有A（或B）确认c。

    • 3NF
      全部函数依赖。3NF在2NF的基础之上，消除了非主属性对于码的传递函数依赖。在码（A)->B, B->C, 则出现了传递函数依赖，此时应该分表为 ab, bc；

    • BC范式
      主码的传递函数依赖。在3NF的基础上，消除了主属性对码的部分与传递函数依赖。举例：在码（A，B，C）->D 中，A->B，此时应该分表为AB，AC->D;

      1人不分解、2人抱团、3人传递、BC抱团者独立。

  2. 事务、问题、隔离级别
    逻辑上的一组操作，要么都执行，要么都不执行。
    事务的ACID特性。

    • atomicity 原子性
    • consistency 一致性 业务相关、程序员规定。比如A给B转钱，一致性的含义就是AB总钱不变。
    • isolation 隔离性 各个事务之间是隔离开来、互不影响的。
    • durability 持久性 对数据的改变是持久的，会落到存储介质上。
      原子性，隔离性和持久性是数据库的属性，而一致性（在 ACID 意义上）是应用程序的属性。

    事务拥有隔离性，但事务并发仍会带来问题。

    脏 读-读到脏数据，脏数据指别人刚改但没提交的。
    丢失修改- B改了A改的数据
    不可重复读- 对一条记录，对第一次和第二次之间有别的事务改动，
    幻读 - 对同一张表，读到了别人新增或者删除的改动

    针对这些问题（脏读、不可重复读、幻读），定义以下隔离级别

    • 读取未提交 可以读没提交的数据，问题最大
    • 读取已提交 没提交的不能读，避免了脏读。但是已提交的可能会影响自身，导致不可重复读和幻读。
    • 可重复读 保证多次读同一条是一样的，但是不能避免幻读（条目增加/减少）。
    • 可串行化
      前三者由MVCC实现，可串行化由锁实现。mysql默认隔离级别是可重复读。

    行锁-根据索引进行锁，没命中索引的话会锁表。

  4. 索引
    底层结构：

    • hash索引
      利用哈希函数进行精准定位，所以快。也因此不能范围查找。还会有hash冲突。
    • B树/B+
      b树的所有节点都会存储key和data（单个查询快），B+树只有叶子节点会存储data,其他节点存储key。
      b+树叶子节点相邻间是有指针的，方便遍历。

    待确认：为什么Mongodb索引用B树，而Mysql用B+树? - 孤独烟的文章 - 知乎
    https://zhuanlan.zhihu.com/p/107228878

    类型：
    聚集索引（数据和索引放在叶子节点、更新代价大主键一般不可变）、
    非聚集索引（索引和数据是分开存放的，比如二级索引叶子节点存一个主键，但没有数据，需要回表查询数据。所以更新代价较小）、

    覆盖索引：要查的索引都有：一个索引包含所有需要查询的字段的值
    最左匹配原则：联合索引（多个字段）使用时，按最左的匹配（遇到范围就停）
