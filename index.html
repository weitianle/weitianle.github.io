Java并发编程笔记（一）

并发编程是为了更快，但有些时候并发不比串行快。

理解并发，就要优先对比串行和并发的差异。

1.上下文切换

单核也有多线程并发，cpu分配非常短的时间片（几十ms）给每个线程，每个线程执行完就要切，所以会频繁地切换。任务从被保存到再次被加载的过程，就叫上下文切换。对于java来说，每次线程从waiting到runnable都会进行一次上下文切换。

如何减少上下文切换？

  • 无锁并发编程，多线程竞争锁的话也会上下文切换
  • cas算法
  • 使用最少线程，避免大量线程处于waiting状态
  • 协程，单线程里的多任务调度。

2. 死锁

常用避免方法：

  • 避免一个线程获取多个锁
  • 避免一个线程在锁内同时占用多个资源
  • 尝试使用定时锁，lock.tryLock(timeout) 来代替内部锁
  • 对于数据库锁，加锁和解锁操作在同一个连接中

资源限制

  • 硬件限制：带宽速度、硬盘读写速度、cpu处理速度

  • 软件限制：数据库连接数、socket连接数

  • 解决：

    • 硬件层面：加机器：集群
    • 软件层面：连接池复用
    • 根据软硬件限制动态调整并发度

常用http状态码

分类：

1. 信息类 100预请求成功，可继续。 101切换协议
  1. 成功 200成功，201创建了资源。202接受但没处理完。204处理成功但无返回。206成功处理部分。
  2. 重定向 301已永久移动到新地址，302临时移动到新地址。304请求的资源未修改，服务器不返回资源，可继续用缓存。305代理。
  3. 客户端。400 客户端语法错误，服务器无法理解。401 未登录，要身份验证。403 拒绝执行。404找不到资源。405 方法被禁止。
  4. 服务器端 500服务器内部错误，无法完成请求。常见异常。 501不支持功能。502网关错误。 503超时。504网关超时。505 http协议超时。

这类码，需要经常接触才能熟悉。

Regular Expression 正则表达式 (一)

正则表达式用于字符串查找和匹配。此篇用于基础，后续会更新进阶。

先自行思考接触过的正则表达式，然后再根据下文查漏补缺。

元字符

\d 匹配数字

\w 匹配字母或数字或下划线或汉字

\s 匹配空白字符

^ 开始符号

$ 结束符号

\b 匹配单词开始或结束

重复限定符

. 匹配任意一个字符(换行符除外)

\ 反斜杠转义

？ 匹配0次或一次

* 匹配0次或多次

+ 匹配1次或多次

{n} 重复n次

{n, } 重复n次或更多次

{n, m} 重复n-m次

分组

（） 小括号括起来为一组

或

｜ eg: ^(189 | 180 | 177)\d{8} $

区间

[0-9]、[a-z]、[].

进阶知识点

零宽断言

断言： 指定内容前后会有满足规则的内容。

零宽：没有宽度，指指匹配位置，不匹配断言本身。

即零宽返回值都不包含以下例子中的pattern。

正向先行

?= pattern， 匹配pattern前面的内容。

正向后行

?<=pattern, 匹配pattern后面的内容

负向先行

?!pattern， 匹配非pattern的前面内容。

负向后行

?<!pattern, 匹配非pattern的后面内容。

捕获和非捕获

捕获一般与分组联系在一起。

数字编号捕获组

(exp) 表达式从左往右，遇到左括号就分一个组。第0组为整个表达式，但不计入分组数量。

(0\d{2})-(\d{8})

命名编号捕获组

(?exp) , 可以用命名来区分组。

(?\0\d{2})-(?\d{8})

非捕获组

(?:0\d{2})-(\d{8})

反向引用

\捕获组编号

基础正则表达式没有记忆能力，所以要用捕获和引用才能记住之前搜过的。

eg: 先捕获一个组，根据数字编号就可以再次用它。

(\w)\1

结束语

数据库

  1. 三范式
    数据库第一二三范式到底在说什么？ - 刘慰的文章 - 知乎
    https://zhuanlan.zhihu.com/p/20028672

    • 1NF
      一个字段表示一个值。不能再分解。比如进货单价和进货数量 必须分开两个字段。

    • 2NF
      部分函数依赖,2NF在1NF的基础之上，消除了非主属性对于码的部分函数依赖。即假设选择码(A, B）- > C ，则不能直接有A（或B）确认c。

    • 3NF
      全部函数依赖。3NF在2NF的基础之上，消除了非主属性对于码的传递函数依赖。在码（A)->B, B->C, 则出现了传递函数依赖，此时应该分表为 ab, bc；

    • BC范式
      主码的传递函数依赖。在3NF的基础上，消除了主属性对码的部分与传递函数依赖。举例：在码（A，B，C）->D 中，A->B，此时应该分表为AB，AC->D;

      1人不分解、2人抱团、3人传递、BC抱团者独立。

  2. 事务、问题、隔离级别
    逻辑上的一组操作，要么都执行，要么都不执行。
    事务的ACID特性。

    • atomicity 原子性
    • consistency 一致性 业务相关、程序员规定。比如A给B转钱，一致性的含义就是AB总钱不变。
    • isolation 隔离性 各个事务之间是隔离开来、互不影响的。
    • durability 持久性 对数据的改变是持久的，会落到存储介质上。
      原子性，隔离性和持久性是数据库的属性，而一致性（在 ACID 意义上）是应用程序的属性。

    事务拥有隔离性，但事务并发仍会带来问题。

    脏 读-读到脏数据，脏数据指别人刚改但没提交的。
    丢失修改- B改了A改的数据
    不可重复读- 对一条记录，对第一次和第二次之间有别的事务改动，
    幻读 - 对同一张表，读到了别人新增或者删除的改动

    针对这些问题（脏读、不可重复读、幻读），定义以下隔离级别

    • 读取未提交 可以读没提交的数据，问题最大
    • 读取已提交 没提交的不能读，避免了脏读。但是已提交的可能会影响自身，导致不可重复读和幻读。
    • 可重复读 保证多次读同一条是一样的，但是不能避免幻读（条目增加/减少）。
    • 可串行化
      前三者由MVCC实现，可串行化由锁实现。mysql默认隔离级别是可重复读。
      怎么解决幻读？
      可串行化/表锁/next-key lock(行锁+间隙锁)

    • B树/B+

  2. tcp三次握手和四次挥手过程
    三次握手才可以阻止重复历史连接的初始化（主要原因）
    三次握手才可以同步双方的初始序列号
    三次握手才可以避免资源浪费

    保证客户端发送的最后一个ack报文段能够到达服务器。因为这最后一个ack确认包可能会丢失，然后服务器就会超时重传第三次挥手的fin信息报，然后客户端再重传一次第四次挥手的ack报文。如果没有这2msl，客户端发送完最后一个ack数据报后直接关闭连接，那么就接收不到服务器超时重传的fin信息报(此处应该是客户端收到一个非法的报文段，而返回一个RST的数据报，表明拒绝此次通信，然后双方就产生异常，而不是收不到。)，那么服务器就不能按正常步骤进入close状态。那么就会耗费服务器的资源。当网络中存在大量的timewait状态，那么服务器的压力可想而知。

    原因2：在第四次挥手后，经过2msl的时间足以让本次连接产生的所有报文段都从网络中消失，这样下一次新的连接中就肯定不会出现旧连接的报文段了。也就是防止我们上一篇文章 为什么tcp是三次握手而不是两次握手？ 中说的：已经失效的连接请求报文段出现在本次连接中。如果没有的话就可能这样：这次连接一挥手完马上就结束了，没有timewait。这次连接中有个迷失在网络中的syn包，然后下次连接又马上开始，下个连接发送syn包，迷失的syn包忽然又到达了对面，所以对面可能同时收到或者不同时间收到请求连接的syn包，然后就出现问题了。

3. 翻转二叉树

二叉树解题的思维模式分两类：

1、是否可以通过遍历一遍二叉树得到答案？如果可以，用一个 traverse 函数配合外部变量来实现，这叫「遍历」的思维模式。

2、是否可以定义一个递归函数，通过子问题（子树）的答案推导出原问题的答案？如果可以，写出这个递归函数的定义，并充分利用这个函数的返回值，这叫「分解问题」的思维模式。

无论使用哪种思维模式，你都需要思考：

如果单独抽出一个二叉树节点，它需要做什么事情？需要在什么时候（前/中/后序位置）做？其他的节点不用你操心，递归函数会帮你在所有节点上执行相同的操作。

5. 层序遍历

void levelTraverse(TreeNode root) {
    if (root == null) return ;
    Queue<TreeNode> q = new LinkedList<>();
    q.offer(root);
    while (!q.isEmpty()) {
        int size = q.size;
        for (int i=0; i<size; i++) {
            TreeNode cur = q.poll();
            //处理节点
            if (cur.left != null) q.offer(cur.left);
            if (cur.right != null) q.offer(cur.right);
        }

    }
}
