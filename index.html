二叉树解题的思维模式分两类：

1、是否可以通过遍历一遍二叉树得到答案？如果可以，用一个 traverse 函数配合外部变量来实现，这叫「遍历」的思维模式。

2、是否可以定义一个递归函数，通过子问题（子树）的答案推导出原问题的答案？如果可以，写出这个递归函数的定义，并充分利用这个函数的返回值，这叫「分解问题」的思维模式。

无论使用哪种思维模式，你都需要思考：

如果单独抽出一个二叉树节点，它需要做什么事情？需要在什么时候（前/中/后序位置）做？其他的节点不用你操心，递归函数会帮你在所有节点上执行相同的操作。

操作系统

操作系统基本概念

  1. 操作系统管理计算机硬件和软件资源的程序，屏蔽了硬件层的复杂性。可理解为人机之间的接口。负责文件管理、进程管理、设备管理、内存管理。

  2. 进程在系统上的运行分为用户态和系统态（内核态）。用户态权限低，系统态要求更高权限。从用户态申请转到内核态的场景，包括系统调用、发生异常、外部设备中断。用户态转内核态时，要保存用户态的寄存器信息，返回时恢复，开销较大。

  3. 进程和线程的区别：
    进程是计算机中的程序关于某数据集合上的一次运行活动，是系统进行资源分配和调度的基本单位。线程是更小的运行单位，是CPU调度和分派的基本单位。一个进程可以产生多个线程，线程共享进程内存空间，线程也有自己的空间（程序计数器、虚拟机栈、本地方法栈）

  4. 进程状态包括 创建（new）-> 就绪(ready) -> 运行(running) -> 阻塞(waiting) ->结束（terminated）（正常或中断）

  5. 进程间通信
    概念：进程间相互独立，数据无法直接交换，所以要找中间人内核。在内核里面开辟缓冲区，进程间通过缓冲区进行通信。

  2. tcp三次握手和四次挥手过程
    三次握手才可以阻止重复历史连接的初始化（主要原因）
    三次握手才可以同步双方的初始序列号
    三次握手才可以避免资源浪费

    保证客户端发送的最后一个ack报文段能够到达服务器。因为这最后一个ack确认包可能会丢失，然后服务器就会超时重传第三次挥手的fin信息报，然后客户端再重传一次第四次挥手的ack报文。如果没有这2msl，客户端发送完最后一个ack数据报后直接关闭连接，那么就接收不到服务器超时重传的fin信息报(此处应该是客户端收到一个非法的报文段，而返回一个RST的数据报，表明拒绝此次通信，然后双方就产生异常，而不是收不到。)，那么服务器就不能按正常步骤进入close状态。那么就会耗费服务器的资源。当网络中存在大量的timewait状态，那么服务器的压力可想而知。

    原因2：在第四次挥手后，经过2msl的时间足以让本次连接产生的所有报文段都从网络中消失，这样下一次新的连接中就肯定不会出现旧连接的报文段了。也就是防止我们上一篇文章 为什么tcp是三次握手而不是两次握手？ 中说的：已经失效的连接请求报文段出现在本次连接中。如果没有的话就可能这样：这次连接一挥手完马上就结束了，没有timewait。这次连接中有个迷失在网络中的syn包，然后下次连接又马上开始，下个连接发送syn包，迷失的syn包忽然又到达了对面，所以对面可能同时收到或者不同时间收到请求连接的syn包，然后就出现问题了。

数据库

  1. 三范式
    数据库第一二三范式到底在说什么？ - 刘慰的文章 - 知乎
    https://zhuanlan.zhihu.com/p/20028672

    • 1NF
      一个字段表示一个值。不能再分解。比如进货单价和进货数量 必须分开两个字段。

    • 2NF
      部分函数依赖,2NF在1NF的基础之上，消除了非主属性对于码的部分函数依赖。即假设选择码(A, B）- > C ，则不能直接有A（或B）确认c。

    • 3NF
      全部函数依赖。3NF在2NF的基础之上，消除了非主属性对于码的传递函数依赖。在码（A)->B, B->C, 则出现了传递函数依赖，此时应该分表为 ab, bc；

    • BC范式
      主码的传递函数依赖。在3NF的基础上，消除了主属性对码的部分与传递函数依赖。举例：在码（A，B，C）->D 中，A->B，此时应该分表为AB，AC->D;

      1人不分解、2人抱团、3人传递、BC抱团者独立。

  2. 事务、问题、隔离级别
    逻辑上的一组操作，要么都执行，要么都不执行。
    事务的ACID特性。

    • atomicity 原子性
    • consistency 一致性 业务相关、程序员规定。比如A给B转钱，一致性的含义就是AB总钱不变。
    • isolation 隔离性 各个事务之间是隔离开来、互不影响的。
    • durability 持久性 对数据的改变是持久的，会落到存储介质上。
      原子性，隔离性和持久性是数据库的属性，而一致性（在 ACID 意义上）是应用程序的属性。

    事务拥有隔离性，但事务并发仍会带来问题。

    脏 读-读到脏数据，脏数据指别人刚改但没提交的。
    丢失修改- B改了A改的数据
    不可重复读- 对一条记录，对第一次和第二次之间有别的事务改动，
    幻读 - 对同一张表，读到了别人新增或者删除的改动

    针对这些问题（脏读、不可重复读、幻读），定义以下隔离级别

    • 读取未提交 可以读没提交的数据，问题最大
    • 读取已提交 没提交的不能读，避免了脏读。但是已提交的可能会影响自身，导致不可重复读和幻读。
    • 可重复读 保证多次读同一条是一样的，但是不能避免幻读（条目增加/减少）。
    • 可串行化
      前三者由MVCC实现，可串行化由锁实现。mysql默认隔离级别是可重复读。
      怎么解决幻读？
      可串行化/表锁/next-key lock(行锁+间隙锁)

为了推迟一些高开销的对象初始化操作，并且只有在使用时才进行初始化。此时采用延迟初始化。

错误示范：

A线程在new 这一步时，B也getInstance,此时虽然不为null, 但是还没初始化完成，B获得了一个未完成初始化的实例。

why? 因为对象创建分为以下三步：

  • 分配内存空间
  • 初始化
  • 将引用变量指向对象地址

其中 初始化和 引用指向地址可能会发生重排序，当先赋值引用时，此时拿到的对象实例并没有完成初始化。

那要怎么解决呢？

  • 防止上面的重排序
  • 重排序可以，但是要做完，不让别的线程看到中间过程。

为防止重排序，可用volatile和sychronized结合。

适用于对实例字段和静态字段的线程安全延迟初始化，优选实例字段。

如果允许重排序，则必须保证重排序的两个操作都执行完，再让其他线程看到。

此方法利用类的初始化会获取锁的机制，完成对类的初始化。此方法简短，适用于对静态字段的线程安全延迟初始化。

该过程可以理解为先拿到锁的线程A，会修改状态标记，然后释放锁再初始化，初始化完成后再拿锁修改状态标记为已完成初始化，再释放锁。其他线程拿到锁也要判断状态标记，如果已经在初始化了或初始化完成了，释放掉锁。

线程A初始化过程中都要两次加锁解锁，初始化完成后其他线程再来就只要拿一次锁判定状态了。

（此处疑问？类的加载、初始化，不是在最开始就执行了吗？为什么要等到多线程来调用才会进行初始化？）

（答案：除了被jvm指定的启动类，其他都是用的时候再开始初始化。包括：

1、创建类的实例,也就是new一个对象
2、访问某个类或者接口的静态变量,或者对该静态变量赋值
3、调用类的静态方法
4、反射(Class.forName(MyClass.class))
5、初始化一个类的子类(初始化子类时,会优先初始化父类)
6、JVM启动时表明的启动类,即文件名和类名相同的那个类）

juc-charpter3-JMM总结

JMM定义了Java 虚拟机(JVM)在计算机内存(RAM)中的工作方式。

JMM内存模型和处理器内存模型都会参考顺序一致性内存模型，但都会放松限制。

JMM基于不同处理器内存模型插入不同的内存屏障指令，所以不同处理器平台展示给程序员的会是一致的内存模型。

JMM内存模型保证以下可见性：

  • 单线程程序
  • 正确同步的多线程程序，保证结果跟顺序一致性内存模型的执行结果一致。
  • 未正确同步的多线程程序，JMM会提供最小安全性保障。线程会读到之前写过的值，或者默认值0、null、false。

Java 内存模型

1.1 并发编程模型的两个关键问题

  • 线程间如何通信？
  • 线程之间如何同步？

通信：共享内存（主内存中写-读内存的公共状态）或者消息传递（显式）

同步：线程之间操作发生相对顺序的机制。共享内存方式下是显式指定顺序，消息传递是隐式。

Java采用共享内存模型，隐式通信。

1.2 Java内存模型的抽象结构

内存模型

堆内存：实例域、静态域和数组元素，线程之间共享。

局部变量、方法定义参数、异常处理器参数不会在线程之间共享，所以没有内存可见性问题。

（JMM） java 内存模型控制java线程之间的通信，线程之间的共享变量存储在主内存中，每个线程都有一个私有的本地内存，本地内存中存储了该线程以读/写变量的副本，本地内存并不真实存在，抽象概念。

通信

从A本地内存到主内存，再到B本地内存的过程，可以称为通信。

重排序：提升效率

​ 1）编译器优化的重排序，在不改变单线程的程序语义的前提下。

​ 2）（处理器级别）指令级并行的重排序，不存在数据依赖性。

​ 3）（处理器级别）内存系统的重排序，利用缓存和读/写缓冲区

通过编译器重排规则和处理器层的内存屏障指令来禁止相应类型的重排序。

处理器中会有写缓冲区（读内存，写缓冲），处理器将要写进内存的数据先放入缓冲区，避免处理器继续等待和内存通信。但每个写缓冲区，仅对自己可见。写缓冲区中的数据也会被处理，批处理刷新写缓冲区，对同一内存地址的写要合并，此时会发生重排序。

（此处便是一个多机缓存的场景）

因为缓冲区的存在，所以处理器的读/写顺序，和内存中的读/写顺序不一定一致。

内存屏障指令

共4类Barriers, 由store、load两个组成，分别是loadload/ loadstore / storeload/ storestore.

load为读，数据装载；store为写，写数据可见即刷新到缓存。

确保（）的顺序在（）之前。

happens-before

一个操作执行的结果需要对另一个操作可见。

  • 程序顺序规则 （按书写顺序）
  • 监视器锁规则（先解锁再加锁）
  • volatile变量 （先写再读）
  • 传递性 （a-b,b-c –> a-c）

但是不代表执行顺序。只是结果可见性。

重排序

单线程

  • 有数据依赖关系的操作不会被重排序。（指单线程，多线程机器不考虑）

  • as-if-serial, 被重排序的单线程执行也不会改变原来的结果。

控制依赖：会猜测为真计算并存入重排序缓冲，若为真则写入。

多线程控制依赖重排序会影响结果，单线程不会。

顺序一致性

是理论参考模型，如果程序正确同步，程序的执行将具有顺序一致性，即执行结果和顺序一致性内存模型中的执行结果相同。

内存模型：

  • 一个线程中的所有操作必须按照程序的顺序来执行
  • 不管程序是否同步，所有线程都只能看到一个单一的操作执行顺序，每个操作原子执行并立刻对所有线程可见。
JMM不保证顺序一致性。临界区内的代码可以重排序。

最小安全性：要么已经被写过，要么初始化为（0,NULL,False）。分配对象时，先内存清零，然后分配。分配时就会完成默认初始化。

volatile的内存语义

可见性：读前都能看到最后的写。

原子性：对单个volatile变量读/写具有原子性。但复合操作没有，如i++。

内存语义：写线程对读线程发了一条消息。

过程：写线程先把本地内存中的共享变量值写到主内存，读线程先把本地内存中的共享变量值置为无效，再从主内存读。

JMM实现：

​ 基于保守策略插入内存屏障实现，禁止重排序。

​ 策略：在读前或写后都加一个storeload屏障

volatile用法

条件：1. 对变量的写不依赖当前值 2. 其他变量不变式不含该变量

常用：

  • 状态标记
  • 一次性安全发布(线程安全或有效不可变的对象)
  • 独立观察的状态标记（传感器）
  • volatile bean， 属性是volatile的能够保证类对象线程安全
  • 开销较少的读写锁策略：读多写。少可用。sychronized(写)和volatile(读)

内存语义：a解锁 b加锁的过程实际上可以看成a对b发了消息。

    ​ 存储java对象信息的地方

    对象头长度：数组3字，非数组2字。

    长度 内容 说明
    32/64 Mark Word 存储对象的hashcode或锁信息
    Class Metadata Address 存储到对象类型数据的指针
    Array length 对象数组长度（如果是数组的话）

    对象头存储结构：

    其中有mark word。用于存储hashcode 分代年龄、和锁标记位。

    锁状态 25bit 4bit 1bit 是否偏向锁 2bit 锁标志位
    无锁 对象hashcode 对象分代年龄 0 0

  3. 具体锁的过程？

    • 首先是偏向锁： 大多数情况下，都是同一个线程多次进入同步块。为了降低获得锁的消耗，使用偏向锁。

      • 过程：

        • 线程进入同步块前：

          栈帧创建空间用来存锁记录，将对象头里的信息暂存到栈帧中。用CAS将线程id放进对象头和栈帧锁记录中。然后进入同步块执行代码。

        • 再次进入：

          不需要CAS加锁和解锁了，只要对比对象头里的线程id和栈帧里的线程id是否一致：

          • 若一致：直接进
          • 若不一致，检查偏向锁是否设置成了1。如果设置了，则用CAS修改对象头里的线程id。如果没有，则用CAS竞争锁。

        • 用完了，要撤销：

          • 竞争出现了才会撤销，不然对象头里还是会存当前线程id.
          • 过程：
            • 先暂停拥有偏向锁的线程，检查它是否活动，不活动则恢复无锁。若活动，说明此时发生竞争。但偏向锁的初衷就是同一个线程多次进入，这里就不再适用偏向锁了。所以要么偏向其他线程，要么恢复无锁或标记该线程不适合作为偏向锁。
            • 唤醒暂停的线程。

      • 可以不用，设置jvm参数。此时就会默认进入轻量级锁。

    • 来到轻量级锁

      • 过程：

        • 线程进入同步块前：

          栈帧创建空间，存储锁记录。对象头的mark word先复制到锁记录中。然后CAS尝试将mark word替换为指向栈帧中锁记录的指针 ，若成功则拿到锁，否则有人竞争，开始自旋。

          自旋：会消耗cpu, 锁升级成重量级锁后，其他线程不会再自旋获取锁，会直接阻塞。只有锁释放后才会唤醒，重新争夺。

        • 解锁：

          • CAS将栈帧中的锁记录替换回对象头，如果成功则没有竞争，解锁成功。否则表示有竞争，锁膨胀成重量级锁，且不会再恢复成轻量级锁。
          • mark word指向互斥量，标记为重量级锁，不再指向锁记录地址。
