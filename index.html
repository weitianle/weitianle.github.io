final 域的内存语义

重排序规则：

  • 写：写final和引用赋值不能重排序。
    • 实现：禁止final重排序到构造函数之外。在写后插入storestore屏障，再接构造函数的return。
  • 读：初次读包含final的引用和读final，不能重排序。
    • 确保先读引用，再读final。final读前插入loadload操作。

  • 程序顺序

双重检查锁定

为了推迟一些高开销的对象初始化操作，并且只有在使用时才进行初始化。此时采用延迟初始化。

错误示范：

A线程在new 这一步时，B也getInstance,此时虽然不为null, 但是还没初始化完成，B获得了一个未完成初始化的实例。

why? 因为对象创建分为以下三步：

  • 分配内存空间
  • 初始化
  • 将引用变量指向对象地址

其中 初始化和 引用指向地址可能会发生重排序，当先赋值引用时，此时拿到的对象实例并没有完成初始化。

那要怎么解决呢？

  • 防止上面的重排序
  • 重排序可以，但是要做完，不让别的线程看到中间过程。

为防止重排序，可用volatile和sychronized结合。

适用于对实例字段和静态字段的线程安全延迟初始化，优选实例字段。

如果允许重排序，则必须保证重排序的两个操作都执行完，再让其他线程看到。

此方法利用类的初始化会获取锁的机制，完成对类的初始化。此方法简短，适用于对静态字段的线程安全延迟初始化。

该过程可以理解为先拿到锁的线程A，会修改状态标记，然后释放锁再初始化，初始化完成后再拿锁修改状态标记为已完成初始化，再释放锁。其他线程拿到锁也要判断状态标记，如果已经在初始化了或初始化完成了，释放掉锁。

线程A初始化过程中都要两次加锁解锁，初始化完成后其他线程再来就只要拿一次锁判定状态了。

（此处疑问？类的加载、初始化，不是在最开始就执行了吗？为什么要等到多线程来调用才会进行初始化？）

（答案：除了被jvm指定的启动类，其他都是用的时候再开始初始化。包括：

1、创建类的实例,也就是new一个对象
2、访问某个类或者接口的静态变量,或者对该静态变量赋值
3、调用类的静态方法
4、反射(Class.forName(MyClass.class))
5、初始化一个类的子类(初始化子类时,会优先初始化父类)
6、JVM启动时表明的启动类,即文件名和类名相同的那个类）

juc-charpter3-JMM总结

JMM定义了Java 虚拟机(JVM)在计算机内存(RAM)中的工作方式。

JMM内存模型和处理器内存模型都会参考顺序一致性内存模型，但都会放松限制。

JMM基于不同处理器内存模型插入不同的内存屏障指令，所以不同处理器平台展示给程序员的会是一致的内存模型。

JMM内存模型保证以下可见性：

  • 单线程程序
  • 正确同步的多线程程序，保证结果跟顺序一致性内存模型的执行结果一致。
  • 未正确同步的多线程程序，JMM会提供最小安全性保障。线程会读到之前写过的值，或者默认值0、null、false。

Java 内存模型

1.1 并发编程模型的两个关键问题

  • 线程间如何通信？
  • 线程之间如何同步？

通信：共享内存（主内存中写-读内存的公共状态）或者消息传递（显式）

同步：线程之间操作发生相对顺序的机制。共享内存方式下是显式指定顺序，消息传递是隐式。

Java采用共享内存模型，隐式通信。

1.2 Java内存模型的抽象结构

内存模型

堆内存：实例域、静态域和数组元素，线程之间共享。

局部变量、方法定义参数、异常处理器参数不会在线程之间共享，所以没有内存可见性问题。

（JMM） java 内存模型控制java线程之间的通信，线程之间的共享变量存储在主内存中，每个线程都有一个私有的本地内存，本地内存中存储了该线程以读/写变量的副本，本地内存并不真实存在，抽象概念。

通信

从A本地内存到主内存，再到B本地内存的过程，可以称为通信。

重排序：提升效率

​ 1）编译器优化的重排序，在不改变单线程的程序语义的前提下。

​ 2）（处理器级别）指令级并行的重排序，不存在数据依赖性。

​ 3）（处理器级别）内存系统的重排序，利用缓存和读/写缓冲区

通过编译器重排规则和处理器层的内存屏障指令来禁止相应类型的重排序。

处理器中会有写缓冲区（读内存，写缓冲），处理器将要写进内存的数据先放入缓冲区，避免处理器继续等待和内存通信。但每个写缓冲区，仅对自己可见。写缓冲区中的数据也会被处理，批处理刷新写缓冲区，对同一内存地址的写要合并，此时会发生重排序。

（此处便是一个多机缓存的场景）

因为缓冲区的存在，所以处理器的读/写顺序，和内存中的读/写顺序不一定一致。

内存屏障指令

共4类Barriers, 由store、load两个组成，分别是loadload/ loadstore / storeload/ storestore.

load为读，数据装载；store为写，写数据可见即刷新到缓存。

确保（）的顺序在（）之前。

happens-before

一个操作执行的结果需要对另一个操作可见。

  • 程序顺序规则 （按书写顺序）
  • 监视器锁规则（先解锁再加锁）
  • volatile变量 （先写再读）
  • 传递性 （a-b,b-c –> a-c）

但是不代表执行顺序。只是结果可见性。

重排序

单线程

  • 有数据依赖关系的操作不会被重排序。（指单线程，多线程机器不考虑）

  • as-if-serial, 被重排序的单线程执行也不会改变原来的结果。

控制依赖：会猜测为真计算并存入重排序缓冲，若为真则写入。

多线程控制依赖重排序会影响结果，单线程不会。

顺序一致性

是理论参考模型，如果程序正确同步，程序的执行将具有顺序一致性，即执行结果和顺序一致性内存模型中的执行结果相同。

内存模型：

  • 一个线程中的所有操作必须按照程序的顺序来执行
  • 不管程序是否同步，所有线程都只能看到一个单一的操作执行顺序，每个操作原子执行并立刻对所有线程可见。
JMM不保证顺序一致性。临界区内的代码可以重排序。

最小安全性：要么已经被写过，要么初始化为（0,NULL,False）。分配对象时，先内存清零，然后分配。分配时就会完成默认初始化。

volatile的内存语义

可见性：读前都能看到最后的写。

原子性：对单个volatile变量读/写具有原子性。但复合操作没有，如i++。

内存语义：写线程对读线程发了一条消息。

过程：写线程先把本地内存中的共享变量值写到主内存，读线程先把本地内存中的共享变量值置为无效，再从主内存读。

JMM实现：

​ 基于保守策略插入内存屏障实现，禁止重排序。

​ 策略：在读前或写后都加一个storeload屏障

volatile用法

条件：1. 对变量的写不依赖当前值 2. 其他变量不变式不含该变量

常用：

  • 状态标记
  • 一次性安全发布(线程安全或有效不可变的对象)
  • 独立观察的状态标记（传感器）
  • volatile bean， 属性是volatile的能够保证类对象线程安全
  • 开销较少的读写锁策略：读多写。少可用。sychronized(写)和volatile(读)

锁

可以让临界区的代码互斥执行，也能让线程间传递消息。

内存语义：a解锁 b加锁的过程实际上可以看成a对b发了消息。
实现

​ 借助于AQS，AQS使用一个整型的volatile state变量记录锁状态。

reentrantlock分为公平锁和非公平锁。公平锁会直接读state然后写state，非公平要用cas竞争锁。

CAS同时具有volatile读 和 写 的内存语义，因为CAS会禁止当前指令和前后的读写指令重排序，并且会将写缓冲的所有数据全部刷到内存里。

通用化的concurrent包：

  • 声明变量为volatile

  • 使用CAS的原子条件更新来实现线程之间的同步

  • 配合volatile的读/写和CAS具有的volatile读和写的内存语义

2.1 volatile

  1. volatile最关键的特性：可见性

    可见性、防止指令重排序。

    可见性：多线程共享变量，一个线程修改时，另一个线程也能读到。

    ​ 实现：用lock前缀指令会引起处理器缓存回写到内存，此时通过处理器缓存一致性协议保证一致性。

    特点：无需加锁，允许线程访问共享变量，线程用排他锁获取。但使用volatile不需加锁，保证可见性，无需上下文切换。‘

    性能优化：追加字节

2.2 synchronized的实现原理

  1. 属性：锁

  2. 锁谁？

    • synchronized void method() {} 普通同步方法 锁当前实例
    • synchronized static void method() {} 静态同步方法 锁类Class对象
    • synchronized(Object) {}。同步方法块（代码块），括号里是谁就锁谁
    1. 怎么锁的？

    都可以用monitor对象实现的，monitorenter 和monitorexit配套使用。拿到monitor即拿到锁。

    锁存在哪里？

    ​ 在java对象头。

    ​ 那对象头是什么？

    ​ 存储java对象信息的地方

    对象头长度：数组3字，非数组2字。

    长度 内容 说明
    32/64 Mark Word 存储对象的hashcode或锁信息
    Class Metadata Address 存储到对象类型数据的指针
    Array length 对象数组长度（如果是数组的话）

    对象头存储结构：

    其中有mark word。用于存储hashcode 分代年龄、和锁标记位。

    锁状态 25bit 4bit 1bit 是否偏向锁 2bit 锁标志位
    无锁 对象hashcode 对象分代年龄 0 0
    轻量级锁 指向栈中锁记录的指针(30bit) 00
    重量级锁 指向互斥量的指针(30) 10
    GC标记 空 11
    偏向锁 线程id、epoch(用时间戳表示有效性) 对象分代年龄 01

  3. 具体锁的过程？

    • 首先是偏向锁： 大多数情况下，都是同一个线程多次进入同步块。为了降低获得锁的消耗，使用偏向锁。

      • 过程：

        • 线程进入同步块前：

          栈帧创建空间用来存锁记录，将对象头里的信息暂存到栈帧中。用CAS将线程id放进对象头和栈帧锁记录中。然后进入同步块执行代码。

        • 再次进入：

          不需要CAS加锁和解锁了，只要对比对象头里的线程id和栈帧里的线程id是否一致：

          • 若一致：直接进
          • 若不一致，检查偏向锁是否设置成了1。如果设置了，则用CAS修改对象头里的线程id。如果没有，则用CAS竞争锁。

        • 用完了，要撤销：

          • 竞争出现了才会撤销，不然对象头里还是会存当前线程id.
          • 过程：
            • 先暂停拥有偏向锁的线程，检查它是否活动，不活动则恢复无锁。若活动，说明此时发生竞争。但偏向锁的初衷就是同一个线程多次进入，这里就不再适用偏向锁了。所以要么偏向其他线程，要么恢复无锁或标记该线程不适合作为偏向锁。
            • 唤醒暂停的线程。

      • 可以不用，设置jvm参数。此时就会默认进入轻量级锁。

    • 来到轻量级锁

      • 过程：

        • 线程进入同步块前：

          栈帧创建空间，存储锁记录。对象头的mark word先复制到锁记录中。然后CAS尝试将mark word替换为指向栈帧中锁记录的指针 ，若成功则拿到锁，否则有人竞争，开始自旋。

          自旋：会消耗cpu, 锁升级成重量级锁后，其他线程不会再自旋获取锁，会直接阻塞。只有锁释放后才会唤醒，重新争夺。

        • 解锁：

          • CAS将栈帧中的锁记录替换回对象头，如果成功则没有竞争，解锁成功。否则表示有竞争，锁膨胀成重量级锁，且不会再恢复成轻量级锁。
          • mark word指向互斥量，标记为重量级锁，不再指向锁记录地址。

Java并发编程笔记（一）

并发编程是为了更快，但有些时候并发不比串行快。

理解并发，就要优先对比串行和并发的差异。

1.上下文切换

单核也有多线程并发，cpu分配非常短的时间片（几十ms）给每个线程，每个线程执行完就要切，所以会频繁地切换。任务从被保存到再次被加载的过程，就叫上下文切换。对于java来说，每次线程从waiting到runnable都会进行一次上下文切换。

如何减少上下文切换？

  • 无锁并发编程，多线程竞争锁的话也会上下文切换
  • cas算法
  • 使用最少线程，避免大量线程处于waiting状态
  • 协程，单线程里的多任务调度。

父亲在脑海里盘算起来，但是一个下午的耕作已经让他的脑袋里由汗液粘成了一团浆糊，做工，免费，这些词他一概也没听清，只是想着自己的儿子能学到一手木工手艺，以后可以代替皮尔切在村子里接活，等多干几年熟练以后，说不定会把生意做到村外，做到市里也不一定。那他们世世代代种谷子的命运算是在他这断绝了，没有比这更好的交易，他装模作样地思索了一阵后同意了，浑然不觉其实这样只是免费卖掉了一个快要身强力壮的劳力，甚至比昨天卖掉的那只五十比索的老年母牛还要便宜。在几年后，他开始感到对农活力不从心而腰酸背痛的无数个日夜里，他经常会这么想“我的儿子甚至不如一只五十比索的牛，就这样被我卖给了皮尔切。该死的皮尔切，该死的厄运，只会欺负我这样靠双手吃饭的可怜人。”这样的想法一直持续到我失踪不见，持续到他的不如意的老年。
在这个村子里，没有人会刻意养花，这是瑞玛王朝里最穷也最微不足道的一个村子，如果把瑞玛王朝广阔的土地看成是一个鸡蛋，那这个村子就是鸡蛋下半部壳上一个最普通不过的气孔，可能是一群旅人依着小溪建立起一个营寨，但要知道，在长途跋涉中最能阻挡人脚步的可能不是突然出现的强盗野兽，也不是有了上顿没下顿的饥荒，而是突如其来的放松和休息，几个旅人扎营建寨，就着篝火的光亮回忆往昔，劣质酒精像一把锤头，锤散了他们的斗志，从此认命，想着这个地方又靠近小溪，又是平原，离一个规模不大不小的镇子路途也不远，酒精和火光烘托出一种宿命感，不如就当定居下来的第一群人吧。
鸡蛋王朝的下半部亮起一丝光亮，又很快变成了一户，几户，十几户，几十户。可能会继续扩大，也可能会很快熄灭。在这天高皇帝远的地方，一个普通地主可能比住在鸡蛋北部，从不踏出皇宫之外的王更有势力。忠诚只是一种宗教，人们左右盘旋，看看哪个宗教供奉的神更能给予精神的安定。
这个村没有人会养花，村民把对生活的热情全部献祭给了活着的每一时刻：种地，学一门手艺，干活，煮饭，生孩子……一点不剩。
当然，除了我，我是吉秋，我见过一株金盏花被月亮照亮的样子。

我是逃离
很多人都会在生活的某个时刻想到我，可是很多人也都会误会我。
我乔装成很多种样子：一阵由近及远的马蹄声，一顿还没有下锅的晚饭，一段暴躁如雷的争吵，一句突如其来的抱怨……我喜欢潜伏在这些瞬间，可是我没什么耐心，很快我又会离开，因为我知道这里面的大部分人都缺乏勇气。

（突然的灵感：逃离认为有勇气的人它会给予他们一点什么，这本书可以误导为找到圣都布尔乔彻底改变命运或者改变这个时代，但其实逃离给予的是主角在路上的各种细节：和朋友的信任，争吵，微醺的夜晚，对明天的希冀，一路看到的美景……这就是逃离给予他的，全新的生活。毕竟到了布尔乔后，什么都不会发生，这是这个时代注定的结局，个人的努力挽不回整个时代的落后。）

常用http状态码

分类：

1. 信息类 100预请求成功，可继续。 101切换协议
  1. 成功 200成功，201创建了资源。202接受但没处理完。204处理成功但无返回。206成功处理部分。
  2. 重定向 301已永久移动到新地址，302临时移动到新地址。304请求的资源未修改，服务器不返回资源，可继续用缓存。305代理。
  3. 客户端。400 客户端语法错误，服务器无法理解。401 未登录，要身份验证。403 拒绝执行。404找不到资源。405 方法被禁止。
  4. 服务器端 500服务器内部错误，无法完成请求。常见异常。 501不支持功能。502网关错误。 503超时。504网关超时。505 http协议超时。

这类码，需要经常接触才能熟悉。

参考：https://www.runoob.com/http/http-status-codes.html

Regular Expression 正则表达式 (一)

正则表达式用于字符串查找和匹配。此篇用于基础，后续会更新进阶。

先自行思考接触过的正则表达式，然后再根据下文查漏补缺。

元字符

\d 匹配数字

\w 匹配字母或数字或下划线或汉字

\s 匹配空白字符

^ 开始符号

$ 结束符号

\b 匹配单词开始或结束

重复限定符

. 匹配任意一个字符(换行符除外)

\ 反斜杠转义

？ 匹配0次或一次

* 匹配0次或多次

+ 匹配1次或多次

{n} 重复n次

{n, } 重复n次或更多次

{n, m} 重复n-m次

分组

（） 小括号括起来为一组

或

｜ eg: ^(189 | 180 | 177)\d{8} $

区间

[0-9]、[a-z]、[].

进阶知识点

零宽断言

断言： 指定内容前后会有满足规则的内容。

零宽：没有宽度，指指匹配位置，不匹配断言本身。

即零宽返回值都不包含以下例子中的pattern。

正向先行

?= pattern， 匹配pattern前面的内容。

正向后行

?<=pattern, 匹配pattern后面的内容

负向先行

?!pattern， 匹配非pattern的前面内容。

负向后行

?<!pattern, 匹配非pattern的后面内容。

捕获和非捕获

捕获一般与分组联系在一起。

数字编号捕获组

(exp) 表达式从左往右，遇到左括号就分一个组。第0组为整个表达式，但不计入分组数量。

(0\d{2})-(\d{8})

命名编号捕获组

(?exp) , 可以用命名来区分组。

(?\0\d{2})-(?\d{8})

非捕获组

(?:0\d{2})-(\d{8})

反向引用

\捕获组编号

基础正则表达式没有记忆能力，所以要用捕获和引用才能记住之前搜过的。

eg: 先捕获一个组，根据数字编号就可以再次用它。

(\w)\1

结束语

正则内容暂时就写到这，以前也只学过简单内容，没想到还有进阶内容。熟能生巧，正则这玩意儿还是得靠多多练习。
